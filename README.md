# Auto GDPR Reject

A Safari Web Extension that automatically rejects GDPR cookie-consent pop-ups
by clicking "Reject All" (or the closest equivalent) for you.

## How it works

The whole extension is one content script, [content.js](extension/content.js),
which runs in every page and frame and:

1. Looks for the reject button of ~20 known consent-management platforms
   (OneTrust, Cookiebot, Didomi, Usercentrics, Osano, CookieYes, Klaro,
   iubenda, HubSpot, Complianz, …) via specific CSS selectors, including
   inside open shadow DOM.
2. Falls back to generic matching: any visible button/link whose label
   exactly matches a known "reject" phrase in ~30 languages ("Reject all",
   "Alle ablehnen", "Tout refuser", "Rechazar todo", …) — but only when the
   button sits inside something that looks like a consent banner, so it never
   clicks "Decline" in an unrelated app dialog.
3. Handles two-step banners: when no direct reject button exists anywhere,
   it clicks "Manage cookies" / "Cookie settings" to open the preferences
   layer and rejects there. If that layer has no reject button either, it
   saves the untouched defaults ("Confirm my choices", "Save preferences") —
   GDPR requires optional categories to start out OFF, so saving the
   defaults equals rejecting.
4. Watches the DOM for late-loading banners with a MutationObserver, allows
   up to 4 clicks (manage step + confirmation layers), and shuts itself
   down after 30 seconds to stay cheap.

## Layout

- [extension/](extension/) — the actual web extension (manifest + content
  script + icons). Edit here.
- `Auto GDPR Reject/` — the Xcode wrapper project generated by Apple's
  `safari-web-extension-converter`. It **references** the files in
  `extension/`, so edits there are picked up on the next build.
- [tools/make_icons.py](tools/make_icons.py) — regenerates the icons.

## Build & install

```sh
cd "Auto GDPR Reject"
DEVELOPER_DIR=/Applications/Xcode.app/Contents/Developer \
  xcodebuild -scheme "Auto GDPR Reject" -configuration Release \
  -derivedDataPath build CODE_SIGN_IDENTITY="-" build
rm -rf "/Applications/Auto GDPR Reject.app"
ditto "build/Build/Products/Release/Auto GDPR Reject.app" \
  "/Applications/Auto GDPR Reject.app"
```

Keep exactly ONE copy of the app on disk (`/Applications`). Safari gets
confused when the same extension is registered from multiple build
locations (e.g. an old `build/` folder plus Xcode's DerivedData), and may
silently hide it from the Extensions list. If you build from Xcode instead,
delete the `/Applications` copy first — or stick to the commands above.

## Enable in Safari

The build is ad-hoc signed (no Apple Developer account), so Safari treats it
as an unsigned extension:

1. Run the app once: `open "/Applications/Auto GDPR Reject.app"`
2. In Safari: **Settings → Advanced → check "Show features for web developers"**.
3. **Develop menu → Allow Unsigned Extensions** (this requires an admin
   password and SILENTLY RESETS every time Safari fully quits — redo it
   after any Safari restart, or the extension disappears from the list).
4. **Safari Settings → Extensions → enable "Auto GDPR Reject"**, and grant it
   access to all websites ("Always Allow on Every Website").

Then visit any EU news site — the cookie banner should vanish on its own
within a second of appearing.

## Releasing (signed + notarized, for Homebrew)

Requires a paid Apple Developer Program membership (for a "Developer ID
Application" certificate). One-time setup is documented at the top of
[scripts/release.sh](scripts/release.sh). Then:

```sh
TEAM_ID=XXXXXXXXXX ./scripts/release.sh 1.0
```

This archives, signs with Developer ID, notarizes, staples, and produces
`dist/Auto-GDPR-Reject-<version>.zip` plus its sha256. Upload the zip to a
GitHub release tagged `v<version>`, then update version + sha256 in the cask
([homebrew/auto-gdpr-reject.rb](homebrew/auto-gdpr-reject.rb)), which lives in
a `homebrew-tap` repo on your GitHub account. Users install with:

```sh
brew tap chadwishner/tap
brew install --cask auto-gdpr-reject
```

Signed + notarized builds do NOT need the "Allow Unsigned Extensions" dance —
the extension shows up in Safari Settings → Extensions unconditionally.

## Iterating

After changing anything in `extension/`, rebuild (step above), then in Safari
use **Develop → Web Extension Background Content** or simply toggle the
extension off/on in Settings → Extensions to reload it.
